# Requirements Document: device-discovery

## Introduction

Device Discovery is the foundation of KeyRx's "True Blank Canvas" philosophy. Rather than assuming keyboard layouts based on OS conventions (QWERTY, AZERTY, ISO, ANSI), this feature allows users to discover their physical keyboard layout by pressing keys in order, creating a per-device profile that maps scan codes to physical positions (row, column).

**Core differentiator**: While other remappers inherit OS assumptions about keys, KeyRx treats every keyboard as a fresh grid of buttons. This enables truly layout-agnostic configurations that can be shared across any keyboard type.

## Alignment with Product Vision

From `product.md`:
> "The keyboard is treated as a pure grid of buttons with physical positions, not OS-defined key names. Through device discovery, each physical key is mapped by its position (row, column), eliminating assumptions about what a key 'should' be."

From `tech.md`:
> "scan_code is generated by keyboard firmware, same across OS. Represents physical button position on keyboard matrix. This is our path to true blank canvas."

This spec implements the Device Discovery Flow defined in the steering documents:
1. Detect keyboard by (vendor_id, product_id)
2. Prompt user to discover physical layout
3. User presses keys in order: top-left → right, then next row
4. Each key registered by physical position: KEY_0_0, KEY_0_1, etc.
5. Profile saved per-device for future sessions

## Requirements

### REQ-1: Device Detection

**User Story:** As a user, I want KeyRx to automatically detect connected keyboards by their hardware identifiers, so that each device can have its own profile.

#### Acceptance Criteria

1. WHEN a keyboard is connected THEN the system SHALL detect its vendor_id and product_id
2. WHEN the system starts THEN it SHALL enumerate all connected input devices
3. WHEN a device is detected THEN the system SHALL check if a profile exists for (vendor_id, product_id)
4. IF no profile exists THEN the system SHALL prompt the user to run discovery or use default fallback
5. WHEN multiple keyboards are connected THEN the system SHALL track each independently by device_id

### REQ-2: Discovery Wizard CLI

**User Story:** As a power user, I want a CLI-based discovery wizard that guides me through mapping my keyboard layout, so that I can set up KeyRx headlessly.

#### Acceptance Criteria

1. WHEN `keyrx discover` is run THEN the system SHALL list detected keyboards and prompt selection
2. WHEN `keyrx discover --device <vendor:product>` is run THEN the system SHALL start discovery for that device
3. WHEN discovery starts THEN the system SHALL instruct: "Press keys from top-left to right, then next row"
4. WHEN a key is pressed during discovery THEN the system SHALL:
   - Record the scan_code
   - Assign position (current_row, current_col)
   - Display: "KEY_0_0 (scan: 0x1) registered"
   - Advance to next column
5. WHEN Enter/Return is pressed THEN the system SHALL advance to the next row
6. WHEN `keyrx discover --save` is complete THEN the system SHALL save the profile to devices/ directory
7. IF the user presses Escape THEN the system SHALL abort discovery without saving
8. WHEN discovery completes THEN the system SHALL display a summary of all registered keys

### REQ-3: Profile Storage

**User Story:** As a user, I want my keyboard profiles persisted, so that I don't have to re-discover on every startup.

#### Acceptance Criteria

1. WHEN a profile is saved THEN it SHALL be stored at `~/.config/keyrx/devices/<vendor>_<product>.json`
2. WHEN saving THEN the profile SHALL include:
   - vendor_id, product_id, name
   - rows count, cols_per_row array
   - keymap (scan_code → position + alias)
   - discovered_at timestamp
3. WHEN KeyRx starts THEN it SHALL load all profiles from the devices/ directory
4. IF a profile is corrupted THEN the system SHALL log a warning and use default fallback
5. WHEN a user runs `keyrx devices list` THEN the system SHALL display all saved profiles

### REQ-4: scan_code as Primary Identifier

**User Story:** As a developer, I want scan_code to be the canonical key identifier in profiles, so that mappings work consistently across operating systems.

#### Acceptance Criteria

1. WHEN processing key events THEN the system SHALL extract scan_code from the event
2. WHEN storing key mappings THEN the key identifier SHALL be scan_code (not OS keycode)
3. WHEN looking up physical key info THEN the query SHALL be by scan_code
4. IF scan_code is unavailable (synthetic events) THEN the system SHALL fall back to OS keycode with warning
5. WHEN exporting profiles THEN scan_codes SHALL be stored as hex strings (e.g., "0x1E")

### REQ-5: Position-Based Naming

**User Story:** As a user, I want keys named by physical position (KEY_0_0, KEY_2_5), so that my configs are layout-agnostic.

#### Acceptance Criteria

1. WHEN a key is discovered THEN it SHALL be assigned name format `KEY_{row}_{col}`
2. WHEN rendering key names THEN the system SHALL use the position-based name by default
3. WHEN a Rhai script references KEY_r_c THEN the system SHALL resolve it to the correct scan_code
4. WHEN displaying unknown keys THEN the system SHALL fall back to `SCAN_{hex}` format
5. WHEN comparing positions THEN row 0 SHALL be top row, col 0 SHALL be leftmost key

### REQ-6: User Alias System

**User Story:** As a user, I want to define custom aliases for keys (e.g., "LeftPinky", "ThumbCluster"), so that my scripts are readable.

#### Acceptance Criteria

1. WHEN `alias(position, name)` is called in Rhai THEN the system SHALL register the alias
2. WHEN a script references an alias THEN the system SHALL resolve it to the physical position
3. WHEN multiple aliases exist for a key THEN all SHALL be valid references
4. WHEN `keyrx aliases list` is run THEN the system SHALL display all defined aliases
5. WHEN aliases conflict with built-in names THEN the user alias SHALL take precedence
6. WHEN saving config THEN aliases SHALL be persisted in the device profile or script

### REQ-7: Multiple Keyboard Support

**User Story:** As a user with multiple keyboards, I want each to have its own profile and behave independently, so that my gaming keyboard and typing keyboard work differently.

#### Acceptance Criteria

1. WHEN multiple keyboards are connected THEN each SHALL be identified by unique device_id
2. WHEN an event arrives THEN the system SHALL include device_id in the InputEvent
3. WHEN looking up profiles THEN the system SHALL use (vendor_id, product_id) from device_id
4. WHEN routing events to scripts THEN the script SHALL have access to source device info
5. WHEN a profile is missing THEN only that specific device SHALL use the fallback

### REQ-8: Default Fallback Profile

**User Story:** As a new user, I want KeyRx to work immediately without discovery, so that I can try it before committing to setup.

#### Acceptance Criteria

1. WHEN no profile exists for a device THEN the system SHALL use default.json fallback
2. WHEN using fallback THEN keys SHALL be identified by OS key name (CapsLock, A, etc.)
3. WHEN running in fallback mode THEN the system SHALL display a notice recommending discovery
4. WHEN `keyrx discover --generate-default` is run THEN a fallback profile SHALL be created from OS mappings
5. IF default.json doesn't exist THEN the system SHALL create it with standard layout on first run

### REQ-9: Discovery Re-run and Edit

**User Story:** As a user, I want to re-run discovery or edit specific keys, so that I can fix mistakes without starting over.

#### Acceptance Criteria

1. WHEN `keyrx discover --overwrite` is run THEN existing profile SHALL be replaced
2. WHEN `keyrx discover --edit KEY_2_3` is run THEN the system SHALL prompt for new key press at that position
3. WHEN editing THEN only the specified key(s) SHALL be modified
4. WHEN `keyrx discover --from-file layout.json` is run THEN the system SHALL import the profile
5. WHEN a profile is edited THEN the system SHALL update discovered_at timestamp

### REQ-10: Device Naming

**User Story:** As a user with multiple keyboards of the same model, I want to give them unique names, so that I can distinguish them in configs.

#### Acceptance Criteria

1. WHEN a device is first discovered THEN the system SHALL prompt for a friendly name
2. WHEN `keyrx devices rename <vendor:product> "New Name"` is run THEN the name SHALL be updated
3. WHEN displaying devices THEN the system SHALL show friendly name alongside vendor:product
4. IF multiple devices share vendor:product THEN the system SHALL distinguish by device_id suffix
5. WHEN referencing device in scripts THEN both friendly name and vendor:product SHALL be valid

## Non-Functional Requirements

### Code Architecture and Modularity

- **Single Responsibility**: Device detection, profile storage, and discovery wizard are separate modules
- **Modular Design**: Profile format is versioned and extensible
- **Dependency Management**: No platform-specific code in profile logic
- **Clear Interfaces**: DeviceRegistry trait enables mock testing

### Performance

- Profile loading: < 50ms for 10 device profiles
- Device enumeration: < 100ms on startup
- Key event to device lookup: O(1) hash table access
- Profile saving: Non-blocking async I/O

### Security

- Profile files stored in user-writable config directory only
- No network access required for discovery
- Profile JSON validated on load (malformed = rejected)
- No code execution from profile files (data only)

### Reliability

- Corrupted profile files logged and skipped (not crash)
- Missing devices gracefully handled (use fallback)
- Discovery can be interrupted safely (no partial state)
- Atomic profile writes (write temp, then rename)

### Usability

- Discovery wizard provides clear, step-by-step instructions
- Progress shown during discovery (X of Y keys registered)
- Mistakes can be undone (--edit, --overwrite)
- Works headlessly (no GUI required)
